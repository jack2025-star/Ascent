# Technical Documentation Creator

## 核心原则：结论优先与宏观视角
在生成文档时，必须遵循“结论优先”的结构。先明确陈述核心结论或最终结果，再展开解释原理和细节。避免罗列繁琐的代码语法，侧重于“为什么这样做”以及“各部分如何协同工作”。

## 文档分类与规范

### 1. 操作日志 (OPERATION_LOG.md)
*   **用途**：记录配置变更、代码重构的具体步骤与宏观原理。
*   **结构要求**：
    *   **任务目标**：一句话概括本次更新的核心目的。
    *   **核心变更步骤**：按逻辑顺序列出关键操作（如引入模块、配置中间件）。
    *   **宏观原理说明**：解释技术选型背后的架构考量（如同源策略、路径兼容性）。
    *   **维护注意事项**：提醒重启进程、清除缓存等后续操作要点。

### 2. 技术原理文档 (PARAMS.md)
*   **用途**：沉淀系统架构、数据流向及核心参数作用的深度解析。
*   **内容侧重**：
    *   **模块化讲解**：按技术栈（后端 Express、前端 HTML/CSS/JS）拆分文档。
    *   **数据流向图**：描述请求从客户端到服务端再返回的完整闭环。
    *   **核心参数表**：仅保留关键参数的作用说明，移除基础语法教程。
    *   **异常排查方向**：提供针对架构层面的常见错误（如 CORS、端口冲突）解决思路。

## 工作流程

1.  **识别需求**：确认用户需要记录的是“操作步骤”还是“原理说明”。
2.  **提取关键点**：从代码变更中提炼出宏观逻辑（如：通过静态托管解决跨域）。
3.  **撰写草稿**：采用精简、专业的语言，确保文档篇幅适中（单文件建议不超过 500 行）。
4.  **版本控制检查**：确保生成的文档符合 `.gitignore` 中的显式追踪规则。

## 示例模板

### 操作日志模板
```markdown
## 📅 [日期]：[更新主题]
### 🎯 任务目标
[核心结论]
### 💡 宏观原理
[架构级解释]
### ⚠️ 注意事项
[维护指南]
```

### 原理文档模板
```markdown
# [模块名称] 核心原理
## 一、架构概览
[整体职责说明]
## 二、关键配置与数据流
[中间件作用与数据流转路径]
```
