# Express 后端核心原理与数据流

## 一、架构概览
本后端服务基于 **Node.js** 和 **Express** 框架构建，主要职责是处理前端发起的 HTTP 请求，执行用户身份验证逻辑，并返回标准化的 JSON 响应。

### 核心模块作用
*   **express**: 提供路由分发、中间件机制及请求/响应对象封装，简化原生 HTTP 服务器的开发。
*   **cors**: 解决浏览器同源策略限制。通过在响应头中添加特定字段（如 `Access-Control-Allow-Origin`），允许跨域前端页面访问后端 API。

---

## 二、关键配置与中间件原理

### 1. 请求体解析 (`express.json()`)
*   **为什么需要？**：HTTP 协议传输的是原始字节流。该中间件负责拦截请求，识别 `Content-Type: application/json`，并将字节流反序列化为 JavaScript 对象挂载到 `req.body`。
*   **数据流向**：客户端 JSON 字符串 → 中间件解析 → 服务端 `req.body` 对象。

### 2. 跨域支持 (`cors()`)
*   **工作原理**：在服务器响应阶段自动注入 CORS 头部信息。当浏览器检测到这些头部时，会放行原本会被同源策略拦截的跨域请求。

### 3. 静态文件服务 (`express.static`)
*   **核心作用**：将本地文件系统目录映射为 Web 服务器的公开资源目录，实现前后端统一部署。
*   **工作机制**：该中间件会拦截所有未匹配到 API 路由的请求。它根据 URL 路径在指定目录下查找对应文件（如 HTML、CSS、JS），并自动设置正确的 MIME 类型返回给浏览器。
*   **架构优势**：
    *   **消除跨域**：前端页面与后端 API 运行在同一端口下，符合同源策略要求，无需额外的 CORS 配置即可进行数据交互。
    *   **简化部署**：用户只需访问一个 URL 即可完成从页面加载到业务逻辑调用的全过程。
*   **路径处理**：配合 `path.join(__dirname)` 使用，确保在不同操作系统（Windows/Unix）下均能正确解析文件的绝对路径。

## 三、业务逻辑与数据验证

### 1. 模拟数据存储
使用内存数组 `users` 存储凭证。**注意**：在生产环境中，此处应替换为数据库查询，且密码必须经过哈希加密存储。

### 2. 登录验证流程 (`POST /login`)
1.  **参数提取**：通过解构赋值从 `req.body` 获取前端提交的用户名和密码。
2.  **身份匹配**：遍历用户列表，比对用户名与密码是否完全一致。
3.  **响应反馈**：
    *   **成功**：返回 `{ success: true, message: "..." }`。
    *   **失败**：返回 `{ success: false, message: "..." }`。
*   **设计意图**：采用统一的 JSON 响应结构，便于前端进行一致的状态判断和 UI 更新。

---

## 四、服务器生命周期

*   **监听端口**：`app.listen(3000)` 启动 TCP 监听，等待客户端连接。
*   **请求分发**：当请求到达时，Express 根据 URL 路径和 HTTP 方法（GET/POST）将流量路由至对应的处理函数。

---

## 五、常见异常与排查方向

| 现象 | 核心原因 | 解决思路 |
| :--- | :--- | :--- |
| `Cannot find module` | 依赖缺失 | 检查 `package.json` 并执行 `npm install` |
| `req.body is undefined` | 解析中间件缺失 | 确认 `express.json()` 已在路由前注册 |
| `CORS Error` | 跨域被拦截 | 确认后端已启用 `cors()` 中间件 |
| `EADDRINUSE` | 端口冲突 | 检查是否有其他进程占用了 3000 端口 |
